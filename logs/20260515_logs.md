## 한 일

Phase 0 — 뼈대 세팅 완료. 기존에 docs/ 폴더에 PRD, TCD, 기능명세서 등 문서를 먼저 작성해두고 코드를 시작하는 방식이 맞다는 걸 다시 확인했다. 문서가 있으니 settings.py를 짤 때 어떤 앱이 필요한지, allauth 설정이 왜 필요한지 바로 보였다.

커스텀 User 모델(AbstractUser)을 처음부터 만들어두는 것이 중요하다. 나중에 추가하면 migration이 꼬여서 고생한다. 이번에는 계획대로 accounts/models.py를 먼저 작성하고 makemigrations → migrate 순서로 진행해서 깔끔하게 됐다.

whitenoise를 처음부터 넣어둔 것도 잘한 결정이다. Railway에 배포할 때 정적 파일 때문에 admin CSS가 깨지는 문제를 미리 막았다.

## 배운 점

- AUTH_USER_MODEL을 settings에 넣으면 반드시 해당 모델이 먼저 있어야 migrate 가능
- SQLite WAL 모드: 쓰기 성능 개선 + 동시 읽기 허용 (PRAGMA journal_mode=WAL)
- whitenoise: Django 내장 정적파일 서버 대신 사용. production에서 필수

## 다음 Phase

Phase 1 — accounts 앱: 회원가입/로그인/프로필 페이지 구현

---

Phase 1 완료.

accounts 앱 인증 시스템을 커스텀 뷰로 구현했다. allauth를 설치했지만 login/signup은 직접 만든 뷰를 사용한다. allauth URL은 config/urls.py에 fallback으로 포함해둬서 allauth 내부 URL(비밀번호 변경 등)도 쓸 수 있다.

migration 과정에서 배운 것: `auto_now_add=True` 필드를 기존 테이블에 추가할 때는 `null=True`를 함께 줘야 makemigrations가 기본값 없이도 통과된다. 이미 row가 있기 때문에 Django가 기본값을 요구하는데, auto_now_add는 자동 설정이라 interactive 질문이 발생한다.

allauth 버전(0.65+)이 올라가면서 ACCOUNT_LOGIN_ATTEMPTS_LIMIT 같은 설정이 deprecated됐다. 새 방식: `ACCOUNT_RATE_LIMITS = {'login_failed': '5/300s'}`, `ACCOUNT_SIGNUP_FIELDS`.

base.html 네비게이션바에서 닉네임이 있으면 닉네임, 없으면 username을 표시한다. 로그아웃은 POST form으로 처리해서 CSRF 보호를 유지했다.
